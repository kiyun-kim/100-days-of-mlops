## 📅 Day 25: Register, Version, and Manage Model Lifecycle

### 1. Task

- **요구사항**: 실험 단계(Experimentation)에서 검증이 완료된 상이한 스펙의 모델 아티팩트들을 중앙 집중형 중앙 모델 저장소(Model Registry)로 승격 및 통합하여, 모델의 생명주기(Lifecycle)를 프로덕션 배포 레이어와 유기적으로 연동할 수 있는 추적 환경을 수립합니다.

- **목표**:
  1. 중앙 모델 저장소 내에 `fraud-detector` 고유 자산 등록
  2. 최상위 모델 메타데이터 자산에 사양 검증용 `fraud` 키워드가 포함된 Description 주입
  3. `baseline` 실행 엔티티를 레지스트리 최초 버전(Version 1)으로 승격하고 `challenger` 별칭 바인딩
  4. `improved` 실행 엔티티를 차순위 버전(Version 2)으로 승격하고 `champion` 별칭 바인딩

---

### 2. Workflow

```text
[ Experimentation Layer: fraud-detection ]
  ├── Run: baseline (f1: 0.80) ──> [Register] ──> Version 1 (Alias: challenger)
  └── Run: improved (f1: 0.89) ──> [Register] ──> Version 2 (Alias: champion)
                                                        │
                                                        v
                                      [ MLflow Model Registry: fraud-detector ]
                                        └── Central Description: "xFusionCorp fraud..."

```

---

### 3. 해결 과정 (Action & Troubleshooting)

#### 3-1. 모델 레지스트리 초기 등록 및 v1 생성 (Baseline)

1. MLflow UI 대시보드 내부 `fraud-detection` 익스페리먼트 그룹에서 `baseline` 실행 엔티티 상세 페이지로 진입했습니다.
2. 하단 **Logged models (1)** 섹션 내부의 `model` 아티팩트 서브 링크를 트리거하여 등록 웹 프레임워크 팝업을 활성화했습니다.
3. **Model** 컴포넌트 설정을 *Create New Model*로 지정하고 고유 식별자 명세인 `fraud-detector`를 입력 후 생성 단계를 완료하여 `Version 1` 상태를 확보했습니다.

#### 3-2. 모델 레지스트리 버전 확장 및 v2 생성 (Improved)

1. `fraud-detection` 실행 리스트 뷰로 복귀하여 하이퍼파라미터 튜닝이 반영된 `improved` 실행 엔티티 상세 페이지로 재진입했습니다.
2. 동일하게 하단 Logged models (1)의 `model` 인터페이스를 가로채어 모델 등록 팝업을 가동했습니다.
3. 이번에는 신규 생성이 아닌 기존 기틀이 마련된 **`fraud-detector`** 엔티티를 셀렉터에서 지정하여 점진적 버전 업그레이드 조치를 통해 `Version 2` 상태를 수립했습니다.

#### 3-3. UI 거버넌스 데이터 주입 및 트러블슈팅 (Description & Aliases)

1. 상단 글로벌 네비게이션 컨텍스트에서 **[Models]** 허브 페이지로 이동한 뒤 `fraud-detector` 캐비닛 항목을 오픈했습니다.
2. **트러블슈팅 및 정정**: 초기 검증 단계에서 테스트 엔진이 실패(Fail)를 반환하는 현상이 발생했습니다. 분석 결과 하위 버전별 디스크립션 필드가 아닌, 최상위 루트 자산의 **Description** 컨텍스트가 공백 상태인 점을 확인했습니다.
3. 이에 따라 최상위 메인 레이아웃의 Description 레이어 내 **`[Write]`** 액션을 활성화하고 검증 정규식 통과를 위한 문자열을 주입 및 영구 보존 조치했습니다.

```text
# fraud-detector 메인 자산에 바인딩한 최종 Description 컨텍스트 사양
Fraud detection model for xFusionCorp Industries transactions.

```

4. 하단 버전 제어 매트릭스 테이블로 이동하여 릴리스 관리 정책에 의거한 논리적 별칭 명세를 각각 매핑했습니다.

- `Version 1` 컴포넌트 레벨 ──> Aliases 필드에 **`challenger`** 주입 후 저장
- `Version 2` 컴포넌트 레벨 ──> Aliases 필드에 **`champion`** 주입 후 저장

### 4. 핵심 개념 정리

#### 4-1. MLflow Model Registry (모델 레지스트리)

- **전문적 정의**: 독립된 실험(Runs) 영역에서 빌드된 모델 아티팩트들 중 실무 가치가 있는 모델을 식별하여 저장, 버저닝, 단계 전환, 주석 관리를 중앙 집중식으로 수행하는 독립형 인프라 레이어입니다.
  💡 **비유하자면**, **'대형 마트의 중앙 검수 물류창고'**. 각 농가(실험실)에서 개별적으로 비닐봉지에 담아 보낸 수많은 사과 묶음(Runs) 중 품질이 보증된 것들만 선별하여 마트 공용 규격 박스(`fraud-detector`)에 넣고 1호 박스, 2호 박스(Versions)라는 공식 바코드를 붙여 진열대로 내보내는 전문 관리 창고와 같습니다.

#### 4-2. Model Aliases (모델 별칭)

- **전문적 정의**: 특정 모델 버전에 부여하는 가변적 유일 식별 레이블(Label)입니다. 하드코딩된 버전 숫자(v1, v2) 대신 추상화된 포인터 이름을 사용하여 외부 API나 클라이언트가 배포 상태를 동적으로 바라보게 만듭니다.
  💡 **비유하자면**: **'가게 문 앞에 걸어두는 [영업중] / [준비중] 표지판'**. 주방 안에서 요리사(엔지니어)가 1대 사장님(v1)에서 2대 사장님(v2)으로 교체되더라도, 가게 외부 간판 아래의 '대표 주방장' 표지판(`champion`)을 v2에게 옮겨 걸어두면 손님(외부 호출 서버)은 내부 교체 사실을 몰라도 항상 최신 요리를 제공받을 수 있는 구조입니다.

#### 4-3. 실무 관점의 의의 (배포 거버넌스)

- 엔터프라이즈 환경에서 CI/CD 스트림을 결합할 때, CD 파이프라인(ArgoCD, Jenkins 등) 코드가 특정 모델 버전인 `v2`를 직접 바라보도록 하드코딩해 두는 것은 매우 위험한 아키텍처입니다. 모델이 고도화될 때마다 배포 인프라 매니페스트를 매번 수정해야 하기 때문입니다.
- 엔지니어는 `champion`과 `challenger` 같은 별칭 구조를 인프라에 안착시킴으로써, 배포 파이프라인 코드는 오직 `champion`이라는 이름의 엔드포인트만 고정 호출하게 만들고, MLflow UI 내부에서 별칭 포인터만 마우스 클릭 한 번으로 이전(Shift)시켜 수초 만에 트래픽을 안전하게 전환하는 롤링 업데이트 및 카나리 배포 전략을 구사할 수 있게 됩니다.

### 5. 무엇을 배웠는가 (Takeaway)

- **추상화 레이어(Aliases)를 통한 결합도 완화**: 외부 추론(Inference) 서버나 CI/CD 웹훅이 구체적인 모델 버전 숫자(v1, v2)를 직접 호출하게 만들면 인프라 간 강한 결합(Tight Coupling)이 발생한다는 점을 깨달았습니다. 가변 포인터인 Aliases 레이어를 중간에 개입시킴으로써, 인프라 변경 없이 운영 모델을 교체하는 아키텍처 설계 능력을 확보했습니다.

- **실험과 관리 레이어의 엄격한 격리(Isolate)**: 연구원들이 자유롭게 모델을 학습시키고 파라미터를 조절하는 실험(Experiment) 공간과, 실제 운영 환경으로 진출하기 위한 최종 게이트웨이인 레지스트리(Registry) 공간 간의 라이프사이클 분리 필요성을 인식했습니다. 이 격리를 통해 무분별한 모델 파일의 프로덕션 유입을 통제하는 거버넌스의 기본 틀을 체득했습니다.

- **선입선출(FIFO) 기반의 중앙 상태 제어 선언**: MLflow가 모델 버전을 부여할 때 데이터베이스의 시퀀스(Sequence)처럼 순차적 적재 순서를 따른다는 인프라 특성을 이해했습니다. 이는 파이프라인 자동화 구현 시 구버전과 신버전의 레지스트리 등록 타임라인을 엄격히 제어해야만 중앙 인덱스가 꼬이지 않는다는 구조적 교훈을 주었습니다.
