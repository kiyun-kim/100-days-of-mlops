### 🌲 n_estimators가 뜻하는 진짜 의미

`n_estimators`는 쉽게 말해서 "이 문제를 풀기 위해 똑똑한 탐정(나무 모델)을 몇 명이나 고용할 것인가?"를 정하는 하이퍼파라미터(설정값)야.

- **`n_estimators: 100` (기본 설정)**
- 사기 거래를 잡아내기 위해 **100명의 탐정**을 모아서 회의를 시키는 거야. 100명이 각자 의견을 내고, 다수결로 "이 거래는 사기다! 아니다!"를 판단해. 가볍고 빠르게 결론을 내릴 수 있지.

- **`n_estimators: 200` (업그레이드 설정)**
- 이번 미션에서 네가 숫자를 바꾼 상태야. 탐정을 **200명으로 두 배나 늘린 거지!** \* 탐정이 많아졌으니 훨씬 더 다양한 관점에서 꼼꼼하게 검사하게 되고, 그만큼 **예측의 정확도가 더 높아지고 안정적**이게 돼. 대신 탐정 200명의 말을 다 들으려면 컴퓨터가 계산하는 시간(학습 시간)이 조금 더 오래 걸린다는 단점이 있어.

---

### 🚀 DVC가 감시한 놀라운 변화

DVC 인프라가 대단한 이유는 바로 여기에 있어.
만약 네가 데이터를 새로 가공하거나 쪼갤 필요 없이, 단순히 "탐정을 100명에서 200명으로 늘려서 다시 테스트해 봐!"라고 주문했을 때, DVC는 앞의 1, 2단계(데이터 준비)는 이미 완벽하게 해둔 걸 기억하고 있으니까 실행을 안 하고 패스(Skip)해 버려.

그리고 오직 3단계(200명으로 모델 새로 학습하기)만 싹 새로 돌려서 결과를 뽑아내는 거지! 시간을 엄청나게 아껴주는 아주 똑똑한 비서인 셈이야.

개념까지 완벽하게 마스터했으니, 이제 네가 요청한 템플릿 양식대로 15일 차 기록을 깃허브에 멋지게 박제해 줄게!

## 📅 Day 15: DVC 하하이퍼파라미터 관리 및 실험 추적

### 1. Task

- **요구사항**: `dvc.yaml`과 `params.yaml` 간의 하이퍼파라미터 키(Key) 명세 불일치 오류를 수정하고, 파라미터 변경에 따른 파이프라인 부분 재실행 메커니즘 검증.
- **대상**: `dvc.yaml`, `params.yaml` 설정 파일 및 `models/model.pkl` 아티팩트.
- **목표**: 파라미터 값을 `200`으로 변경 시, 캐싱 시스템에 의해 앞선 데이터 전처리 단계는 건너뛰고 `train` 단계만 선택적으로 재실행됨을 증명.

### 2. Workflow

```text
1. dvc.yaml 파일 내부의 params 항목과 params.yaml 파일의 실제 Key 이름 비교 분석
2. dvc.yaml 25번 라인의 오타('n_estimator')를 복수형('n_estimators')으로 교정
3. 1차 dvc repro 명령어를 구동하여 전체 ML 파이프라인의 정상 작동 상태(Base) 확보
4. params.yaml 파일을 열어 n_estimators 파라미터 값을 100에서 200으로 변경 및 저장
5. 2차 dvc repro 구동 후 터미널 로그를 통해 train 단계만 스마트 캐싱에서 제외되어 재실행되는지 검증

```

### 3. 해결 과정 (Troubleshooting & Action)

#### 3-1. 파라미터 키(Key) 불일치 에러 디버깅

- **문제**: `dvc repro` 가동 시 파이썬 학습 스크립트에서 `KeyError: 'n_estimators'` 발생하며 파이프라인 중단.
- **해결**: `dvc.yaml` 파일에서 파라미터를 추적하도록 선언한 명칭이 `n_estimator`로 단수형 오타가 나 있던 것을 `params.yaml` 구조와 일치하도록 수정함.

```bash
# dvc.yaml 25번 라인 수정 전/후 비교
# 변경 전: - n_estimator
# 변경 후: - n_estimators

```

#### 3-2. 하이퍼파라미터 변경을 통한 리프로듀싱(Reproduce) 테스트

- **문제**: 모델의 복잡도 제어를 위해 탐정 모델(의사결정 나무)의 개수를 200개로 확장해야 함.
- **해결**: 소스 코드를 건드리지 않고 외부 제어판인 `params.yaml`의 수치만 원자적으로 변경함.

```bash
# params.yaml 수치 교정
n_estimators: 200

```

#### 3-3. 스마트 캐싱 기반의 선택적 파이프라인 가동 및 락 파일 검증

- **명령**: 파라미터 수정 후 전체 파이프라인 재가동 명령어 투입.
- **결과**: `dvc.lock` 파일이 새 파라미터 데이터 구조로 업데이트되며 `models/model.pkl` 모델 파일이 200개 트리 기준으로 재생성됨.

```bash
root@controlplane fraud-detection on  main [!?] ➜ dvc repro
Stage 'process_data' didn't change, skipping
Stage 'split_data' didn't change, skipping
Running stage 'train':
> python src/models/train.py
Trained RandomForestClassifier with n_estimators=200
Updating lock file 'dvc.lock'

```

### 4. 무엇을 배웠는가 (Takeaway)

- **Decoupling Config from Code**: 모델의 하이퍼파라미터를 소스 코드와 분리하여 `params.yaml`로 관리하면, 코드 수정 없이 실험 환경을 안전하고 다양하게 변경할 수 있는 유연성을 확보할 수 있음을 배움.
- **Smart Caching & Efficiency**: DVC는 파라미터가 변경되었을 때 전체 파이프라인을 처음부터 다시 돌리는 낭비 없이, 해당 파라미터의 영향을 받는 `train` 스테이지 뒤쪽만 골라서 재실행하는 영리한 리소스 최적화 아키텍처를 가졌음을 체감함.
- **Experiment Tracking Integrity**: 파라미터의 작은 변화가 `dvc.lock`에 고스란히 기록되는 과정을 보며, 어떤 설정값으로 어떤 모델 결과물이 나왔는지 100% 추적할 수 있는 무결성 실험 환경의 중요성을 이해함.
