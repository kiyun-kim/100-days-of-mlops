## 📅 Day 28: Fix a Broken MLflow Project and Re-Run It

### 1. Task

- **요구사항**: 어떤 엔지니어가 와도 명령어 한 줄만으로 동일한 환경과 파라미터를 제어하여 머신러닝 학습을 완전하게 재현(Reproduce)할 수 있도록, `MLproject` 청사진 파일의 명세 플래그 버그를 찾아 수정하고 다중 파라미터 실행 실험을 자동화합니다.

- **목표**:
  1. `MLproject` 파일 내 command 실행 플래그를 `train.py` 내부 파싱 규격(`--n_estimators`)에 맞추어 정정
  2. `trainer` 익스페리먼트 공간 내에 `FINISHED` 상태의 성공 런(Run) 2개 이상 확보
  3. 1차 실행(파라미터 200 주입)과 2차 실행(기본값 100 작동)의 결과 메타데이터 유기적 기록

---

### 2. Workflow

```text
[ Developer / Engineer ] ──> Invoke: mlflow run . --env-manager=local
                                    │
                                    v
       +─────────────────────────────────────────────────────────+
       |                  MLproject (청사진 설명서)              |
       |  "python train.py --n_estimators {n_estimators} ..."    |
       +─────────────────────────────────────────────────────────+
                                    │
                                    v (파라미터 주입 및 후킹)
       +─────────────────────────────────────────────────────────+
       |            train.py (실제 작동하는 훈련 엔진)         |
       |  - 1차: n_estimators=200 -> [Success: FINISHED]         |
       |  - 2차: n_estimators=100 -> [Success: FINISHED]         |
       +─────────────────────────────────────────────────────────+

```

---

### 3. 해결 과정 (Action & Troubleshooting)

#### 3-1. 청사진 명세 파일 버그 추적 및 수정

1. 콘솔 로그에서 `train.py: error: unrecognized arguments: --n-estimators 200` 에러 패턴을 탐색했습니다.
2. `train.py` 내부의 `argparse`는 `_` 구조인 `--n_estimators`를 요구하는 반면, 설명서 역할인 `MLproject` 파일에서 `-`인 `--n-estimators`로 어긋나 있는 점을 포착했습니다.
3. `MLproject` 파일 내용을 규칙에 맞추어 정정했습니다.

```yaml
# MLproject 파일 내 command 블록 수정 ('-'을 '_'로 교체)
command: >
  python train.py
  --n_estimators {n_estimators}
  --max_depth {max_depth}
  --test_size {test_size}
  --random_seed {random_seed}
```

#### 3-2. MLflow Run 명령어를 통한 멀티 실험 자동 가동

디렉터리 자산 경로(`/root/code/trainer`)에서 로컬 환경 매니저 플래그를 결합하여 명세서가 요구한 2회의 테스트 런을 가동했습니다.

```bash
# 1차 실행: n_estimators 파라미터를 200으로 명시하여 실행 조치
mlflow run . -e train -P n_estimators=200 -P max_depth=10 --env-manager=local

# 2차 실행: 파라미터를 생략하여 시스템 기본값(100)으로 실행 조치
mlflow run . -e train --env-manager=local

```

- **결과 검증**: 두 명령 모두 `mlflow.projects: === Run (ID '...') succeeded ===` 콘솔 로그를 정상 획득하며 중앙 인프라에 `FINISHED` 레코드로 안착되었습니다.

### 4. 핵심 개념 정리

#### 4-1. MLflow Project (실험 종합 패키지 상자)

- **정의**: 머신러닝 코드를 타인이나 다른 컴퓨팅 환경에서 재실행할 수 있도록 파일 구조, 실행 명령어, 의존성 패키지 사양을 하나의 디렉터리 내에 규격화하여 묶어놓은 표준 패키징 포맷입니다.
  💡 **'레고 블록 조립 설명서가 들어있는 세트 상자'**. 내가 아주 멋진 레고 성(`train.py`)을 만들었습니다. 이걸 친구에게 똑같이 만들어보라고 그냥 블록만 주면 엉뚱한 모양이 나올 수 있습니다. 그래서 상자 안에 "이 부품을 쓰고, 순서는 이렇게 조립해!"라고 적은 [마법의 설명서(MLproject)]를 같이 동봉해서 상자째로 선물하는 것입니다. 그럼 친구는 설명서 첫 페이지를 열자마자 나와 완벽하게 똑같은 레고 성을 순식간에 복제해낼 수 있습니다.

#### 4-2. 명령어 매개변수 바인딩 (Arguments Matching)

- **정의**: `MLproject` 파일의 변수 템플릿 인터페이스(`{n_estimators}`)와 실제 구동 파일(`train.py`)의 명령줄 인수(`--n_estimators`) 간의 텍스트 토큰을 일치시켜 외부 입력값을 내부 프로세스로 유실 없이 전달하는 인프라 바인딩 메커니즘입니다.
  💡 **'자판기 버튼과 음료수 창고 연결하기'**. 실제 기계 안에는 [콜라_창고]라고 언더바 이름표가 붙어있는데, 자판기 바깥 버튼에는 [콜라-버튼]이라고 하이픈 이름표를 달아둔 것입니다. 로봇(컴퓨터)이 바깥 버튼을 보고 안쪽 창고로 찾아가려 하니 "이름이 달라서 콜라*창고를 못 찾겠어!" 하고 멈춘 상황이었습니다. 겉면 버튼 이름표를 똑같이 \*\*[콜라*창고]\*\* 구조로 고쳐주니 로봇이 막힘없이 음료수를 꺼내오기 시작한 것입니다.

#### 4-3. 실무 관점의 의의

- 인공지능 연구원들은 각자 자신의 PC 컴퓨터 환경에서 각기 다른 버전으로 코드를 짜는 경향이 있습니다. 이를 그대로 두면 운영 서버로 모델을 이관할 때 "내 컴퓨터에서는 잘 돌아갔는데 왜 서버에서는 에러가 나지?" 하는 **환경 파편화 문제**가 100% 발생합니다.

- 엔지니어는 전체 코드를 `MLproject`라는 단일 규격 템플릿 상자로 캡슐화해 둠으로써, 인프라 전체의 '재현성(Reproducibility)'을 완벽하게 확보할 수 있습니다. 명령줄 플래그 포인터를 통일했기 때문에 향후 쿠버네티스(Kubernetes) 같은 대규모 클러스터 클라우드 환경에서도 명령어 한 줄(`mlflow run`)만 던지면 인프라 레이어가 알아서 환경을 빌드하고 대량의 자동 튜닝 실험을 안전하게 수행할 수 있는 기반이 됩니다.

---

### 5. 무엇을 배웠는가 (Takeaway)

- **설명서 레이어와 코드 레이어의 싱크(Synchronization) 검증**: 인프라 오케스트레이션 단계에서 에러가 날 때, 실제 연동 프로그램(`train.py`)의 입력 관문(argparse)이 정의한 규격을 원점 명세서와 글자 한 톨 단위까지 크로스체크(RCA)해야 인프라 장애를 빠르게 가둘 수 있음을 학습했습니다.
