## 📅 Day 30: MLflow 모델 서빙(Serving) 가동 및 엔드포인트 라이브 헬스 모니터링 자동화

### 1. Task

- **요구사항**: 실험 파이프라인에서 도출된 최적의 모델 아티팩트를 운영 환경으로 승격(Promotion)시키기 위해 MLflow 기본 서빙 엔진을 활용하여 독자적인 API 엔드포인트 인프라를 가동하고, 지속적인 서비스 가용성 검증을 위한 헬스 프로빙(Probing) 모니터링 스크립트를 안착시킵니다.

- **목표**:
  1. `fraud-detection-v2` 내 최고 F1-Score(0.92) 실행인 `improved`를 `fraud-detector-v2` 모델 저장소의 `champion` 별칭으로 바인딩
  2. **5001번 포트** 및 로컬 환경 매니저를 지정하여 백그라운드 모델 서빙 가동
  3. `/root/code/monitor.sh` 헬스 체크 자동화 스크립트의 유효 경로 배포 및 실행 권한 컴파일

---

### 2. Workflow

```text
[ MLflow Model Registry ] ──> models:/fraud-detector-v2@champion
                                      │
                                      v (mlflow models serve --port 5001)
[ Live Inference Server ] ──────────────────────────────────────────────+
  ├── Uvicorn Web Server 가동 (http://127.0.0.1:5001)
  └── 라우트 개방: /invocations (추론 API), /health (헬스 첵 API)

[ Monitoring Layer ] ───────────────────────────────────────────────────+
  └── monitor.sh ──> 주기적 curl 프로빙 ──> HTTP 200 OK 감지시 exit 0

```

### 3. 해결 과정 (Action & Troubleshooting)

#### 3-1. 최적 실행 식별 및 모델 레지스트리 별칭 매핑

1. MLflow UI에서 `baseline`, `improved`, `regression` 3개 가동 실행의 평가지표를 연계 대조했습니다.
2. `f1_score` 기준 0.92로 최상위 성능을 증명한 **`improved`** 실행을 `fraud-detector-v2` 중앙 저장소에 등록했습니다.
3. 모델 상세 관리 레이어 진입 후, 외부 인프라가 참조할 고유 논리 주소 포인터인 **`champion`** 별칭(Alias)을 해당 버전에 영구 바인딩했습니다.

#### 3-2. 로컬 포트 격리 기반 모델 서빙 프로세스 구동

트래킹 서버 주소 도메인을 선언하고, 5001번 네트워크 소켓을 점유하는 백그라운드 라이브 모델 추론 데몬을 가동했습니다.

```bash
# 1. 트래킹 서버 주소 연계 및 내보내기
export MLFLOW_TRACKING_URI="http://localhost:5000"

# 2. 5001번 포트를 지정하여 백그라운드(&) 서빙 레이어 트리거
mlflow models serve -m "models:/fraud-detector-v2@champion" --port 5001 --env-manager=local &

# 3. 소켓 오픈 상태 검증 (Uvicorn 데몬 정상 바인딩 확인)
ss -ntlp | grep 5001
```

#### 3-3. 모니터링 쉘 스크립트 배치 및 권한 부여

```bash
# 1. 템플릿 파일을 monitor.sh 파일로 복사
cp /root/code/monitor.sh.template /root/code/monitor.sh

# 2. 스크립트 실행 권한 할당
chmod +x /root/code/monitor.sh

# 3. 정정된 작업 공간 절대/상대 경로 기준 쉘 가동 검증
./monitor.sh

```

- **결과 검증**: 터미널 실행 결과에서 `GET /health HTTP/1.1 200 OK` 및 `healthy` 문자가 원격 유실 없이 도출되며 전체 채점 엔진 검증(Check) 레이어를 통과했습니다.

### 4. 핵심 개념 정리

#### 4-1. MLflow Models Serve (모델 서빙 서버)

- **정의**: 학습이 완료되어 저장소에 패킹된 머신러닝 모델 아티팩트를 네트워크 통신(HTTP REST API)이 가능한 웹 서버 인프라 형태로 변환하여, 외부 클라이언트 시스템이 실시간으로 데이터를 던져 예측값을 얻어 갈 수 있도록 가동하는 서빙 호스트 프로세스입니다.
  💡 **'컴퓨터 천재 로봇에게 전용 상담 전화기 놔주기'**. 실험실 구석에서 훈련을 마친 똑똑한 바둑 로봇(모델 파일)이 있습니다. 이 로봇은 혼자 구석에 있으면 아무도 말을 걸 수 없습니다. 그래서 로봇 앞에 [5001번 전화기(포트)]를 설치하고 전원을 켜둔(`mlflow models serve`) 것입니다. 이제 다른 동네 사는 사람들이 이 5001번 번호로 전화를 걸어 "다음 바둑 어디다 둬?"라고 물어보면 로봇이 실시간으로 대답해 줄 수 있는 상태가 된 것입니다.

#### 4-2. 모델 등록(Register)

- **본질**: 머신러닝 파이프라인에서 수많은 실험(Runs)을 돌리다 보면 수십 개의 모델 파일(.pkl 등)이 무분별하게 생성됩니다. 이 중 성능이 좋아서 **실제 서비스(프로덕션)에 배포할 만한 가치가 있는 모델만 선별하여 온전한 '공식 제품'으로 승격시키는 게이트웨이**가 바로 모델 등록(Register)입니다.
  - ##### 4-2-1. 무엇을 위한 과정인가? (목적과 의의)
    - **실험과 운영 환경의 격리**: 데이터 과학자가 마구잡이로 낙서하며 테스트하는 '실험 공간'과 운영 팀이 엄격하게 관리하는 '실업 운영 공간' 사이에 명확한 경계선을 그어줍니다.

    - **추적 가능성(Lineage) 유지**: 모델이 레지스트리에 등록되더라도, 이 모델이 정확히 어떤 실험(Run ID), 어떤 파라미터, 어떤 데이터셋을 통해 태어났는지에 대한 탯줄(연결고리)이 끊어지지 않고 추적됩니다.

    - **안전한 버저닝(Versioning) 관리**: 동일한 이름(`fraud-detector-v2`)으로 모델을 계속 업데이트하더라도 시스템 내부적으로 Version 1, Version 2 형태로 차곡차곡 쌓여 배포 사고 시 즉각적인 롤백을 가능하게 합니다.

    - ##### 4-2-2. 동작 원리

      MLflow UI에서 특정 런을 선택하고 [Register Model]을 누르면 인프라 내부에서는 다음과 같은 기계적 메커니즘이 수행됩니다.

    - **아티팩트 격리 및 이동**: 해당 실험의 디렉터리 깊숙이 숨겨져 있던 모델 바이너리 파일과 환경 설정 파일(`MLmodel`, `conda.yaml`)을 MLflow가 관리하는 중앙 집중형 모델 저장소(Model Registry Cache) 영역으로 복사(또는 소프트 링크 바인딩)합니다.

    - **메타데이터 중앙 집중화**: 중앙 관계형 데이터베이스(PostgreSQL 등)에 해당 모델의 이름, 생성 시간, 등록자 정보, 버전을 기록하는 신규 엔티티를 생성합니다.

    - **추상화 포인터(별칭) 개방**: 등록이 완료되는 순간, 외부에서 구체적인 파일 경로를 몰라도 `models:/모델이름/버전` 혹은 `models:/모델이름@별칭`이라는 단순한 주소(URI)만으로 모델을 원격 호출하여 불러올 수 있는 API 통로를 개방합니다.

    💡
    - **등록 전 (Runs 상태)**: 농부(데이터 과학자)가 밭(실험실)에서 수많은 사과(모델)를 수확하여 바닥에 무더기로 쌓아둔 상태입니다. 어떤 사과가 썩었는지, 어떤 사과가 가장 맛있는지 겉으로는 구별하기 힘듭니다.

    - **등록 과정 (Register Model Action)**: 그 수많은 사과 중 크고 빛나는 특상품 사과를 골라내어, **백화점 공용 규격 박스(`Register`)에 담고 1호, 2호라는 정품 도장(Version)을 쾅 찍어서 백화점 중앙 물류창고(Model Registry) 진열대에 올리는 행동**입니다. 이렇게 상자에 들어가 도장이 찍힌 사과라야 비로소 손님(추론 서버)에게 공식 상품으로 판매(배포)될 수 있습니다.

#### 4-3. Endpoint Health Probing

- **정의**: 마이크로서비스 아키텍처(MSA) 환경에서 특정 서비스 엔드포인트(`http://.../health`)에 주기적으로 가벼운 네트워크 패킷(curl 등)을 유입시켜, 현재 서비스가 클라이언트 요청을 수입할 수 있는 가용(Healthy) 상태인지 중단(Down) 상태인지 상태 코드를 감지하는 인프라 감시 기법입니다.
  💡 **'가게 문이 열렸나 주기적으로 똑똑 문 두드려보기'**. 전화를 설치했는데 기계가 고장 나거나 로봇이 잠들 수 있습니다. 그래서 감시원 스크립트(`monitor.sh`)에게 "너는 5001번 방에 가서 매번 문을 똑똑 두드리고 잘 살아있는지(`200 OK`) 확인해!"라고 임무를 준 것입니다. 문을 두드렸을 때 안에서 "저 잘 있어요!(`healthy`)"라고 대답하면 감시원은 안심하고 보스(인프라 엔진)에게 성공 보고(`exit 0`)를 올리게 됩니다.

### 5. 무엇을 배웠는가 (Takeaway)

- **현재 작업 디렉터리(CWD)와 상대 경로 연산 감각**: 쉘 스크립트 인프라 배포 시 터미널의 현재 세션 절대 위치와 가동 명령어 스트링 간의 매핑 구조를 엄격히 판단해야 경로 오인으로 인한 프로세스 붕괴를 예방할 수 있음을 실증했습니다.

- **프로덕션 릴리스 게이트웨이 파이프라인 수립**: 실험(Experiment) -> 선별(Compare) -> 등록 및 별칭 추상화(Registry) -> 실시간 호스팅(Serving) -> 인프라 모니터링(Health Check)으로 이어지는 **MLOps 전체 라이프사이클 엔드투엔드(End-to-End) 파이프라인**을 내재화하여 주니어 엔지니어로서 실무적인 인프라 시야를 완벽히 구축했습니다.
